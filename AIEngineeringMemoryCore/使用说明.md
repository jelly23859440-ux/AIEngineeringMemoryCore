## 1. 工作原理

```
┌──────────────┐     对话消息      ┌──────────────────┐
│   执行 AI    │ ───────────────→ │  HeartbeatMonitor │
│  (用户对话)   │ ←─────────────── │   Token 累计监控    │
└──────────────┘   注入记忆上下文   └────────┬─────────┘
                                           │ 达到 80% 阈值
                                           ▼
                                  ┌──────────────────┐
                                  │   Consolidator   │
                                  │   调用管理 AI     │
                                  │   压缩对话历史     │
                                  └────────┬─────────┘
                                           │ MemoryPacket
                                           ▼
┌──────────────────────────────────────────────────────┐
│                    .memory/ 本地存储                    │
│  ┌───────────┐  ┌──────────┐  ┌──────┐  ┌─────────┐ │
│  │ summaries │  │ entities │  │ cold │  │ chroma  │ │
│  │ 情景骨架   │  │ 实体索引  │  │ 冷归档│  │ 向量索引  │ │
│  └───────────┘  └──────────┘  └──────┘  └─────────┘ │
└──────────────────────────────────────────────────────┘
```

**核心流程**：

1. 你和 AI 的每一轮对话都经过 [HeartbeatMonitor](file:///d:\AI\Traeproject\AIEngineeringMemoryCore\heartbeat_monitor.py) 累计 token 数
2. 当 token 接近模型上下文上限的 **80%** 时，自动触发"心跳巩固"
3. [Consolidator](file:///d:\AI\Traeproject\AIEngineeringMemoryCore\consolidator.py) 调用管理 AI，将对话历史压缩为 **[MemoryPacket](file:///d:\AI\Traeproject\AIEngineeringMemoryCore\models.py)**（包含情景摘要 + 精确实体）
4. 记忆存入本地 `.memory/` 目录并通过 [ChromaDB](file:///d:\AI\Traeproject\AIEngineeringMemoryCore\memory_retriever.py) 建立向量索引
5. 下一次对话前，[MemoryRetriever](file:///d:\AI\Traeproject\AIEngineeringMemoryCore\memory_retriever.py) 自动检索相关记忆，注入执行 AI 的上下文

## 2. 环境准备

### 2.1 Python 版本

要求 **Python 3.10+**。

```bash
python --version
# 至少显示 Python 3.10.x
```

### 2.2 模型服务（二选一）

#### 方案 A：本地 Ollama（推荐）

```bash
# 安装 Ollama: https://ollama.com/download
# 拉取管理模型
ollama pull llama3

# 拉取嵌入模型（用于向量检索）
ollama pull nomic-embed-text

# 验证服务
curl http://localhost:11434/api/tags
```

#### 方案 B：云端 OpenAI 兼容 API

任何兼容 OpenAI Chat Completions 格式的 API 均可，例如：
- OpenAI 官方 API
- Azure OpenAI
- 本地 vLLM / text-generation-webui
- 其他第三方代理

> **注意**：嵌入模型也需要对应的 API 端点。使用云端 API 时，需确保嵌入端点也配置正确（参见 [配置详解](#5-配置详解)）。

## 4. API 参考

### 4.1 AIMemoryHub 主控类

所有操作通过 [AIMemoryHub](file:///d:\AI\Traeproject\AIEngineeringMemoryCore\memory_hub.py) 完成。

```python
from config import AppConfig
from memory_hub import AIMemoryHub

config = AppConfig()
hub = AIMemoryHub(config, root_dir=".")
hub.start()
```

#### `hub.record_exchange(user_msg: str, ai_response: str) -> None`

记录一轮对话。自动累加 token 计数，达到阈值时触发心跳巩固。

```python
hub.record_exchange(
    user_msg="请帮我实现用户登录模块",
    ai_response="好的，我来创建登录接口，使用 JWT 认证方案。"
)
```

#### `hub.inject_memory_context(query: str) -> str`

根据查询检索相关记忆，返回格式化后的"系统记忆回溯"文本，可直接注入执行 AI 的系统提示词。

```python
context = hub.inject_memory_context("用户登录模块")
if context:
    system_prompt += "\n\n" + context
```

返回示例：

```
[系统记忆回溯 - AI记忆中枢]
- 正在开发电商系统，已完成用户模块，当前在支付模块。 [可信度: high]

相关精确实体：
- payment_gateway = stripe [可信度: high]
- refund_callback_timeout = 5000ms [可信度: medium]
```

#### `hub.retrieve_relevant_memories(query: str) -> str`

与 `inject_memory_context` 相同，检索并格式化记忆上下文。

#### `hub.mark_subtask_complete(task_id: str, keywords: List[str]) -> None`

标记一个子任务完成，将其记忆归档为冷数据。

```python
hub.mark_subtask_complete(
    task_id="payment_module",
    keywords=["支付模块", "Stripe", "退款回调", "payment"]
)
```

之后只有当用户提问命中这些关键词时，相关记忆才会被重新唤醒。

#### `hub.get_memory_protocol() -> str`

返回记忆协作协议文本，应注入到执行 AI 的系统提示词中：

```
[记忆协作协议 - AI记忆中枢]
请遵循以下记忆使用规则：
1. 优先采用标记为 [可信度: high] 的记忆
2. 对 [可信度: medium] 的记忆可参考但自行判断
3. 对 [可信度: low] 或 [manual_corrected] 的记忆谨慎参考
4. 若发现记忆与事实矛盾，明确指出并建议用户更新
5. 需要精确值时，直接引用并说明来源
```

#### `hub.consolidate_pending() -> Optional[MemoryPacket]`

手动触发一次待整理记忆的整合处理（去重、纠错、索引重构）。

#### `hub.get_pending_packets() -> List[MemoryPacket]`

获取当前所有已巩固但未整合的记忆包。

#### `hub.start() / hub.stop()`

启动 / 停止后台调度器和资源。

### 4.2 心跳监控器（HeartbeatMonitor）

```python
from heartbeat_monitor import HeartbeatMonitor

monitor = HeartbeatMonitor(context_limit=4096, heartbeat_ratio=0.8)

# 记录一段文本，返回 token 数
token_count = monitor.track_message("Hello World")

# 注册巩固回调（达到阈值时自动触发）
monitor.register_callback(lambda: print("触发心跳巩固！"))

# 手动强制触发
monitor.force_consolidate()

# 巩固完成后重置计数
monitor.reset()
```

### 4.3 冷热分级管理器（ColdStorageManager）

```python
from cold_storage_manager import ColdStorageManager

cold_mgr = ColdStorageManager(project_manager)

# 标记子任务完成
cold_mgr.mark_subtask_complete("task_001", ["支付", "Stripe"])

# 检查用户查询是否命中冷记忆触发词
triggered = cold_mgr.check_keywords_triggered("支付模块怎么实现？")
# → ["task_001"]

# 激活冷记忆
packet = cold_mgr.activate_cold_memory("task_001")

# 查看所有冷记忆触发词
cold_mgr.get_all_cold_keywords()
# → {"task_001": ["支付", "Stripe"]}
```

### 4.4 数据模型

```python
from models import EntityRecord, EpisodeSummary, MemoryPacket, HeartbeatState

# 实体记录
entity = EntityRecord(
    entity_name="api_base_url",
    entity_type="config",
    exact_value="https://api.example.com/v1",
    confidence="high"
)

# 情景摘要
episode = EpisodeSummary(
    summary_text="完成了用户注册模块的开发，使用邮箱+验证码方案",
    confidence="high",
    related_entity_ids=[entity.entity_id],
    trigger_keywords=["用户", "注册"]
)

# 记忆包

    episodes=[episode],
    entities=[entity]
)

# 序列化
from models import packet_to_json, json_to_packet, packet_to_file, file_to_packet

json_str = packet_to_json(packet)          # → JSON 字符串
restored = json_to_packet(json_str)        # → MemoryPacket
packet_to_file(packet, Path("memory.json"))  # → 写入文件
loaded = file_to_packet(Path("memory.json")) # → 从文件读取
```

## 5. 配置详解

编辑项目根目录下的 [config.yaml](file:///d:\AI\Traeproject\AIEngineeringMemoryCore\config.yaml)：

```yaml
# ─── 管理 AI 配置 ───
# 用于记忆巩固、去重、纠错的 AI 模型
management_ai:
  api_base: "http://localhost:11434/v1"   # API 端点
  api_key: ""                              # API 密钥（本地 Ollama 可留空）
  model: "llama3"                          # 模型名称
  max_retries: 3                           # 失败重试次数
  timeout: 60                              # 请求超时（秒）
  temperature: 0.3                         # 生成温度（越低越精确）

# ─── 记忆存储配置 ───
memory:
  storage_path: "./.memory"                # 记忆存储目录（相对于项目根目录）
  heartbeat_ratio: 0.8                     # 心跳触发阈值比例（80% 即触发）
  context_limit: 4096                      # 执行 AI 的上下文 token 上限
  cold_storage_enabled: true               # 是否启用冷热分级归档

# ─── 记忆检索配置 ───
retrieval:
  top_k_summaries: 5                       # 每次检索返回的摘要数量
  top_m_entities: 10                       # 每次检索返回的实体数量
  embed_model: "nomic-embed-text"          # 嵌入模型名称
  embed_api_base: "http://localhost:11434"  # 嵌入 API 端点（注意没有 /v1）

# ─── 调度器配置 ───
memory_manager:
  mode: "hybrid"                           # 调度模式：async | realtime | hybrid
  async_heartbeat_batch: 60                # 异步模式下，累计多少次心跳后批量整理
```

### 配置项详解

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `management_ai.api_base` | `http://localhost:11434/v1` | 管理 AI 的 API 地址，末尾带 `/v1` |
| `management_ai.model` | `llama3` | 推荐 `llama3`、`qwen2.5`、`mistral` 等中大型模型 |
| `memory.heartbeat_ratio` | `0.8` | 设为 `1.0` 则永不自动触发巩固 |
| `memory.context_limit` | `4096` | 应与执行 AI 的实际上下文窗口一致 |
| `retrieval.embed_model` | `nomic-embed-text` | 轻量嵌入模型，生成 768 维向量 |
| `retrieval.embed_api_base` | `http://localhost:11434` | 嵌入端点**不带 `/v1`**（Ollama 嵌入 API 路径不同） |
| `memory_manager.mode` | `hybrid` | 见下方[调度器三种模式](#9-调度器三种模式) |

## 6. 使用场景

### 场景一：长项目中的上下文接续

```python
hub = AIMemoryHub(config)
hub.start()

# Day 1: 讨论项目架构
hub.record_exchange("我们要开发一个电商系统", "好的，建议采用微服务架构...")
hub.record_exchange("数据库选型?", "推荐 PostgreSQL 做主库，Redis 做缓存...")

# Day 2: 继续开发，AI 自动回忆
context = hub.inject_memory_context("我们之前决定用什么数据库？")
# 输出：
# [系统记忆回溯 - AI记忆中枢]
# - 项目架构讨论：决定采用微服务架构 [可信度: high]
# 相关精确实体：
# - database = postgresql [可信度: high]
# - cache = redis [可信度: high]
```

### 场景二：多子任务切换

```python
# 完成用户模块
hub.mark_subtask_complete("user_module", ["用户", "注册", "登录", "JWT"])

# 切换到支付模块
hub.record_exchange("支付模块怎么集成 Stripe?", "需要先安装 stripe SDK...")

# 几天后回来问用户模块的问题 → 触发词命中，冷记忆自动唤醒
context = hub.inject_memory_context("JWT token 过期时间是多少？")
# 自动从冷归档中唤醒 user_module 的记忆
```

### 场景三：多项目隔离

```python
# 项目 A
hub_a = AIMemoryHub(config, root_dir="./project_a")
hub_a.record_exchange("设计 API 网关", "采用 Kong...")

# 项目 B —— 完全隔离，不会读取到项目 A 的记忆
hub_b = AIMemoryHub(config, root_dir="./project_b")
hub_b.record_exchange("设计数据管道", "采用 Airflow...")

# 两个项目的 .memory/ 目录完全独立
```

### 场景四：与 Trae/Copilot 等 IDE AI 集成

将记忆中枢作为一个后台服务运行，在每次向 IDE AI 提问前：

```python
# 伪代码：在 Trae 对话流程中注入记忆

def before_send_message(user_query):
    """发送消息前的钩子"""
    context = hub.inject_memory_context(user_query)
    protocol = hub.get_memory_protocol()

    # 组合系统提示词
    system_prompt = f"{protocol}\n\n{context}" if context else protocol
    return system_prompt

def after_receive_response(user_query, ai_response):
    """收到回复后的钩子"""
    hub.record_exchange(user_query, ai_response)
```

    │   ├── a1b2c3d4-....json         #   每个摘要一个文件
    │   └── e5f6g7h8-....json
    │
    ├── entities/                     # 精确实体索引（JSON 文件）
    │   ├── i9j0k1l2-....json         #   每个实体一个文件
    │   └── m3n4o5p6-....json
    │
    ├── cold/                         # 冷记忆归档
    │   ├── payment_module.json       #   完整子任务的记忆包
    │   └── user_module.json
    │
    ├── chroma/                       # ChromaDB 向量数据库

**你随时可以**：
- 查看任意 `.json` 文件了解记忆内容
- 备份整个 `.memory/` 目录
- 删除 `.memory/` 目录清空全部记忆

## 8. 冷热分级机制

```
    热记忆 (Hot)                     冷记忆 (Cold)
┌──────────────────┐           ┌──────────────────────┐
│ 日常活跃检索      │           │ 子任务完成后归档       │
│ 每次对话自动匹配   │  ──────→  │ 仅触发词命中时唤醒     │
│ 消耗向量检索算力   │  完成标记  │ 不消耗日常算力         │
└──────────────────┘           └──────────────────────┘
```

**使用建议**：

- 一个子任务开发完成后（如"支付模块"），立即调用 `mark_subtask_complete` 将其归档
- 触发词设置应覆盖该任务的核心概念，例如 `["支付", "Stripe", "退款", "payment"]`
- 如果后续需要频繁访问某个已归档任务，说明不该归档，应等待更合适的时机

## 9. 调度器三种模式

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| **async** | 心跳满 60 次后，后台异步批量整理 | 长对话流，不介意少量延迟 |
| **realtime** | 每次检索前都先整理一次 | 需要最新记忆的场景 |
| **hybrid**（默认） | async 为主 + 紧急呼叫接口 | **推荐**：兼顾性能和时效性 |

```yaml
# config.yaml
memory_manager:
  mode: "hybrid"             # 改为 async 或 realtime
  async_heartbeat_batch: 60  # async 模式下调整批次大小
```

**紧急呼叫**：在 hybrid 模式下，心跳达到 80% 阈值时会立即触发紧急巩固，不等待批次。

## 10. 故障排查

### 10.1 依赖安装失败

```bash
# 如果 chromadb 安装失败（通常因为 Visual C++ Build Tools 缺失）
# 方案1：安装 VS Build Tools
# https://visualstudio.microsoft.com/visual-cpp-build-tools/

# 方案2：使用预编译版本
pip install chromadb --only-binary=:all:
```

### 10.2 "ModuleNotFoundError: No module named 'pydantic'"

```bash
pip install -r requirements.txt
```

### 10.3 管理 AI 调用超时

```
httpx.ReadTimeout: ...
```

解决方法：
1. 检查 Ollama 服务是否在运行：`curl http://localhost:11434/api/tags`
2. 增加超时时间：在 `config.yaml` 中设置 `management_ai.timeout: 120`
3. 检查模型是否已拉取：`ollama list`

### 10.4 嵌入模型调用失败

```
httpx.HTTPStatusError: 404
```

检查配置：
- `retrieval.embed_api_base` 应为 `http://localhost:11434`（**不带 `/v1`**）
- 嵌入模型是否已拉取：`ollama pull nomic-embed-text`

### 10.5 ChromaDB 文件锁定错误（Windows）

```
PermissionError: [WinError 32] 另一个程序正在使用此文件
```

这是 Windows 上 ChromaDB 的已知问题。在代码中调用 `hub.stop()` 后加 `import gc; gc.collect()` 确保资源释放。

### 10.6 记忆没有被检索到

可能原因：
1. ChromaDB 索引尚未建立 → 确保已调用 `retriever.initialize_chroma()`
2. 记忆已被归档为冷数据 → 检查触发词是否匹配
3. 查询文本与记忆内容不相关 → 尝试更精确的查询

### 10.7 清除所有记忆重新开始

```bash
rm -rf .memory/
# 或 PowerShell:
# Remove-Item -Recurse -Force .memory/
```

重启后会自动创建新的空记忆目录。
